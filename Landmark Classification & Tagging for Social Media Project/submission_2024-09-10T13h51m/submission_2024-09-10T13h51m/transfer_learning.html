Question: Outline the steps you took to get to your final CNN architecture and your reasoning at each step. Describe why you think the architecture is suitable for the current problem.

Answer:

  1. Choosing the Base Model: I opted for ResNet16 as the base CNN architecture. The reason is that ResNet, or Residual Networks, are designed to solve the vanishing gradient problem, which can occur when training deep neural networks. By using residual connections, the model ensures that gradients flow through the network during backpropagation, even in very deep layers. This guarantees a more stable training process and better performance, making it a strong candidate for image classification tasks.

  2. Why ResNet16: ResNet16 has a moderate depth, which strikes a balance between complexity and efficiency. With more layers, deeper networks can capture more complex features and patterns in images. However, they can also be prone to overfitting, especially when data is limited. ResNet16 provides enough layers to extract important image details, without the risk of overcomplicating the model. This depth also allows for effective feature extraction and edge detection, which is critical for fine-tuning on the current image dataset.

  3. Pretrained Weights: I initialized the model with pretrained weights, which were trained on a large dataset like ImageNet. The advantage here is that the model has already learned general patterns from "in-the-wild" images, which are similar to the dataset I am fine-tuning it on. This approach significantly reduces training time and increases the models accuracy, as the network doesn't need to start from scratch.

  4. Fine-Tuning: After selecting the ResNet16 architecture, I fine-tuned the model on the specific dataset. This involved unfreezing the final layers and training them on the task-specific data. Fine-tuning helped the model adjust to the particular patterns in the target dataset, while retaining the general image features learned from the pretrained weights.

  5. Hyperparameter Tuning: To optimize performance, I experimented with various hyperparameters:

  6. Learning rate: I started with a smaller learning rate to ensure the pretrained weights werent overwritten too quickly during fine-tuning. Batch size: I tested different batch sizes to balance training time and model performance. Epochs: I monitored the model's performance on the validation set and adjusted the number of epochs to prevent overfitting. Validation and Performance: After hyperparameter tuning, the model was tested on the validation set, achieving a performance over 60%, which was the minimum requirement. Through careful selection of the architecture, fine-tuning, and hyperparameter adjustments, I optimized the model, aiming for 80% or higher accuracy.

from src.predictor import Predictor
from src.helpers import compute_mean_and_std

# First let's get the class names from our data loaders
class_names = data_loaders["train"].dataset.classes

# Then let's move the model_transfer to the CPU
# (we don't need GPU for inference)
model_transfer = model_transfer.cpu()
# Let's make sure we use the right weights by loading the
# best weights we have found during training
# NOTE: remember to use map_location='cpu' so the weights
# are loaded on the CPU (and not the GPU)
model_transfer.load_state_dict(
    torch.load("model_transfer.pt", map_location="cpu")
)

# Let's wrap our model using the predictor class
mean, std = compute_mean_and_std()
predictor = Predictor(model_transfer, class_names, mean, std).cpu()

# Export using torch.jit.script
scripted_predictor = torch.jit.script(predictor)
scripted_predictor.save("transfer_exported.pt")
